## Предметная область

Программа моделирует модуль «Учёт финансов» для банковского приложения ТигрБанк.
Система предназначена для управления личными финансами пользователя и включает:
- создание и управление банковскими счетами (`BankAccount`)
- ведение категорий доходов и расходов (`Category`)
- регистрацию финансовых операций (`Operation`)
- расчёт текущего баланса счёта
- экспорт и импорт всех данных в форматах **CSV**, **JSON** и **YAML**

Модель позволяет учитывать доходы и расходы, связывать операции со счётом и категорией, а также формировать снимок состояния системы для сохранения и восстановления данных.

Таким образом, программа моделирует базовый слой финансового учета внутри банковского приложения.

### Возможные проблемы при расширении функционала

Несмотря на соблюдение принципов SOLID, при дальнейшем расширении системы могут возникнуть следующие сложности:

##### 1. Переход к постоянному хранению данных (БД)

Сейчас используется in-memory хранилище. При добавлении базы данных потребуется:
- новая реализация репозиториев
- настройка ORM
- изменение конфигурации приложения

Однако благодаря использованию интерфейсов репозиториев, замена будет затрагивать только слой инфраструктуры.

##### 2. Добавление новых типов операций

Если появятся новые типы операций (например, перевод между счетами), потребуется:
- расширение `OperationType`
- изменение логики расчёта баланса
- добавление новой бизнес-логики в сервисах

Это повлияет на слой доменной логики.

##### 3. Расширение аналитики

При добавлении сложной аналитики (например, прогнозирование бюджета или финансовая статистика по месяцам):
- потребуется выделение отдельного аналитического слоя
- возможна необходимость агрегационных сервисов
- увеличится нагрузка на архитектуру хранения данных

##### 4. Добавление новых форматов экспорта

Добавление новых форматов (например, XML) потребует:
- введения новой стратегии сериализации
- расширения сервиса импорта/экспорта

Если таких форматов станет много, целесообразно внедрить паттерн Strategy.

#### Почему введённые абстракции улучшают качество дизайна
##### 1. Разделение ответственности (SRP)

Каждый класс отвечает только за одну задачу:
- `BankAccount` — управление состоянием счёта
- `Operation` — описание финансовой операции
- `Repository` — доступ к данным
- `Service` — бизнес-логика

Это делает код:

- предсказуемым
- легко тестируемым
- легко расширяемым

##### 2. Инверсия зависимостей (DIP)

Сервисы зависят от абстракций (интерфейсов репозиториев), а не от конкретных реализаций.

Это позволяет:
- заменить in-memory реализацию на БД без изменения бизнес-логики
- упростить тестирование (можно подменять реализации)

##### 3. Открытость к расширению (OCP)

Архитектура позволяет добавлять:
- новые реализации репозиториев
- новые форматы экспорта
- дополнительные сервисы (аналитика, отчёты)

без изменения существующего кода.

##### 4. Использование Dependency Injection

Spring Boot используется как DI-контейнер.

Преимущества:
- отсутствует ручное создание зависимостей
- упрощено управление жизненным циклом объектов
- код становится модульным и расширяемым

### Архитектурный подход

Приложение построено по принципам слоистой архитектуры:

- `Domain` — сущности предметной области
- `Repository` — абстракции хранения данных
- `Service` — бизнес-логика
- `Console` — демонстрация работы приложения

Такой подход соответствует современным практикам backend-разработки.

### Итог

В результате разработана расширяемая доменная модель финансового учета с соблюдением принципов SOLID и использованием DI-контейнера Spring Boot.

Архитектура обеспечивает:

- модульность
- расширяемость
- тестируемость
- разделение ответственности

Программа может служить базой для дальнейшего развития полноценного банковского финансового модуля.
